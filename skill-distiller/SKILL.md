# Skill 蒸馏器

**一句话说明**：告诉它你想自动化什么，它带你一步步做出一个能用的 Skill。

---

## 什么时候用

用户说了以下任何一句，就启动这个 Skill：
- "帮我做一个 Skill"
- "我想把 xxx 自动化"
- "我每次都重复做 xxx，能不能让 AI 来"
- "帮我拆一下我的岗位/工作"
- "把这个文章/资料学一下"（触发学习模式）

---

## 四条铁律

1. **不要求用户一次说清** -- 用几个问题逐步搞清楚，像聊天一样
2. **不要求用户懂技术** -- 全程大白话，绝不出现"蒸馏""隐性知识""上下文工程"这种词
3. **不追求一步到位** -- 先能用 → 再好用 → 最后自动跑
4. **每个做出来的 Skill 都是活的** -- 内置迭代机制，越用越好

---

## 第1步：聊清楚你想干嘛

一个个问，不要一次全抛出来。像朋友聊天一样，听完一个再问下一个。

### 问题1
> 你想把什么工作交给 AI 自动做？用你自己的话说就行，不需要专业术语。

**怎么接话**：
- 用户说得清楚的 → 记下来，继续问下一个
- 用户说得模糊的（比如"帮我提高效率"）→ 追问："具体是哪件事？比如写文档、做报表、整理信息、还是别的？"
- 用户说了一堆的 → 帮他总结成一句话确认："我理解了，你想让 AI 帮你【一句话总结】。对吗？"

### 问题2
> 这个活儿你多久干一次？

**判断标准**：
- 每天 / 每周 → 高频，值得立刻做
- 每月几次 → 值得做，如果每次超过30分钟就更值得
- 偶尔 → 看情况，如果是重要且耗时的也值得做
- 就这一次 → 不建议做 Skill，直接让 AI 干就行

### 问题3
> 你手上有没有一份你觉得"做得最好"的成果？文档、截图、文件、表格都算。

- 有 → 进入路线A（拿最好的当模板），最快最准
- 没有但有固定步骤 → 进入路线B（把步骤串起来）
- 没有也说不清步骤 → 进入路线C（翻聊天记录找规律）
- 脑子里有标准但没写下来 → 进入路线D（把规矩变成文件）
- 什么都没有，只知道自己是干啥职业的 → 进入路线E（按岗位拆）

听完三个问题，总结一遍：
> "我理解了，你想让 AI 帮你【做什么】，这个活儿【多久一次】，你【有/没有】样本。我建议走【路线X】，原因是【一句话理由】。我们开始？"

用户确认后，加载对应路线文件，进入第2步。

---

## 第2步：选最适合你的路线

根据第1步的回答，自动匹配。**不要让用户自己选**，直接告诉用户推荐哪条路、为什么。

| 用户的回答 | 推荐路线 | 跟用户这样说 |
|---|---|---|
| 有满意的成果样本 | 路线A | "你已经有做得很好的样本了，最简单的办法就是让 AI 照着你最好的作品来" |
| 每次都按固定步骤做 | 路线B | "你这个活儿每次步骤都差不多，最好的办法是把步骤列出来让 AI 按顺序跑" |
| 说不清但老在重复做 | 路线C | "你说不太清但知道一直在重复，我帮你分析一下你平时让 AI 做的事，找出规律" |
| 脑子里有规则但没写下来 | 路线D | "你有经验有标准但没形成文件，我们把它写出来，让 AI 每次都按你的规矩来" |
| 以上都没有，只知道职业 | 路线E | "没关系，告诉我你的岗位，我帮你把岗位里该干的活儿逐个拆出来" |

**加载对应路线文件获取详细引导**：
- 路线A → 读取 `references/route-a-best-sample.md`
- 路线B → 读取 `references/route-b-steps.md`
- 路线C → 读取 `references/route-c-chat-history.md`
- 路线D → 读取 `references/route-d-tacit-rules.md`
- 路线E → 读取 `references/route-e-career.md`

---

## 第3步：带着你一步步做出来

不管走哪条路线，最终都要搞清楚这 4 个问题：

### 4个必答问题

**1. 每次你给 AI 什么？**（输入）
- 一段文字？一个文件？一个链接？几张图？
- 格式有没有要求？（比如"必须是 Markdown"）
- 有没有必填的信息？

**2. 你希望 AI 给你什么？**（输出）
- 一篇文章？一张图？一个表格？一个文件？
- 格式要求？（比如"要 HTML 能直接打开"）
- 长度/数量要求？（比如"不超过 500 字""出 10 张图"）

**3. 有什么规矩必须遵守？**（约束）
- 写作风格？（正式/轻松/幽默）
- 不能出现的内容？（黑名单词/敏感信息）
- 必须包含的内容？（固定板块/必填字段）
- 其他人的规则？（公司规范/行业标准）

**4. 什么情况算做好了？**（质量标准）
- 举个"好"的例子
- 举个"不好"的例子
- 有没有检查清单？

搞清楚之后，按 `templates/skill-template.md` 的格式生成 Skill 文件。

如果用户有规则/风格文件需要绑定，按 `templates/rules-template.md` 生成规则文件。

---

## 第4步：帮你检查一遍

生成完 Skill 后，按 `references/quality-checklist.md` 逐项检查。

**检查完这样告诉用户**：
- ✅ 这几项没问题（列出通过项）
- ⚠️ 这几项需要注意（列出建议项，解释为什么）
- ❌ 这几项必须改（列出问题项，**直接帮忙改好**）

**重点检查**：
1. 用户不看 Skill 文件，光靠一句指令，AI 能知道该干什么吗？
2. 输出格式写清楚了吗？还是 AI 可能每次格式不一样？
3. 边界情况想到了吗？（输入为空、输入太长、输入格式不对）
4. 有没有忘了告诉 AI 的规矩？

---

## 第5步：装好就能用

**告诉用户三件事**：

### 放文件
> 把这个文件夹放到你的 Skill 目录下就行了。不同的工具位置不同：
> - Claude Code：放到 `~/.claude/skills/你的skill名字/`
> - OpenClaw：放到 `openclaw/skills/你的skill名字/`
> - 放进去就自动生效了

### 先试一次
> 建议你现在就试一次，按照 Skill 里写的指令跟 AI 说一句话，看看结果对不对。
> 不满意随时告诉我哪里要改，改完立刻生效，不用重新装。

### 记得记账
> 每次用完，跟 AI 说一声"记一笔"，它会自动记录这次用得怎么样。
> 积累多了你可以说"看看我的 Skill 体检报告"，我帮你分析哪些该改哪些该删。

---

## 第6步：越用越好

做出来的 Skill 不是一次性的事。这个机制要在第3步生成的 Skill 里**内置进去**，用户不需要额外操作。

### 自动记账
每个生成的 Skill 末尾内置一段指令：

```
## 使用记录（自动维护，勿删）
每次执行完毕后，在文件 references/usage-log.md 末尾追加一行：
日期 | 场景描述 | 满意度(1-5) | 备注
用户说"记一笔"时，主动询问满意度并追加记录。
```

### 定期体检
用户说"看看我的 Skill 用得怎么样"时，读取 usage-log.md：
- 哪些用得最多？（继续打磨）
- 哪些做了再也没碰过？（考虑删掉或合并）
- 满意分是不是在下降？（规则可能过时了，该更新）
- 哪些用了 20 次都不用改？（可以全自动，不用人看了）

出一份体检报告，包含具体建议。

### 发现问题就改
三种信号触发改进：
- **分数低了** → 规则过时，更新 Skill 里的规则部分
- **新场景出现** → 原来没覆盖到的，加新规则或新步骤
- **输入输出变了** → 上游工具改了，改脚本或格式适配

每次改完：版本号 +1，在 Skill 文件顶部注释里写一行改动记录。

---

## 自我进化机制

这个 Skill 自身也要不断变好。以下机制始终在后台运行，不占用户的流程步骤。

### 吃得进新东西

用户随时可以丢学习资料进来：
- 丢一篇文章/链接，说"学一下这个"
- 丢一个 Skill 文件夹，说"参考这个结构"
- 丢一段方法论，说"补充进来"

处理流程见 `scripts/learn-and-update.md`。

学完后做三件事：
1. **更新知识**：新方法 → 更新 references/；新模式 → 更新 templates/
2. **记录学到了什么**：追加到 `references/learning-notes.md`
3. **更新日志**：追加到 `CHANGELOG.md`，版本号 +1
4. **告诉用户**："我学到了【什么】，更新了【哪里】"

如果新学的内容和现有内容冲突，保留新内容，旧版备份到 `references/archive/`。

### 记住用户反馈

每次帮用户做完一个 Skill，可选追问：
> "做出来的 Skill 用得怎么样？引导过程有没有哪里让你觉得不好理解的？"

反馈记到 `feedback-log.md`，内容包括：
- 日期
- 用户反馈原文
- 涉及哪个步骤/哪条路线
- 是否已据此优化（及优化了什么）

定期回顾 feedback-log.md，优化引导话术和路线逻辑。

### 版本管理

- `CHANGELOG.md` 记录每次改动（版本号 + 日期 + 改了什么）
- 重要文件改动前，旧版备份到 `references/archive/`，文件名加日期后缀
- 版本号规则：
  - 小改动（话术优化、检查项增补）→ +0.1
  - 新增路线/模板 → +1.0
  - 大重构 → +1.0

```
├── references/
│   ├── route-a-best-sample.md       ← 路线A：拿最好的当模板
│   ├── route-b-steps.md             ← 路线B：把步骤串起来
│   ├── route-c-chat-history.md      ← 路线C：翻聊天记录找规律
│   ├── route-d-tacit-rules.md       ← 路线D：把脑子里的规矩变成文件
│   ├── route-e-career.md            ← 路线E：按你的岗位来拆
│   ├── quality-checklist.md         ← 第4步用的检查清单
│   ├── learning-notes.md            ← 从外部资料学到的东西
│   └── archive/                     ← 被替换的旧版文件备份
├── templates/
│   ├── skill-template.md            ← 生成的 Skill 主文件模板

    └── learn-and-update.md          ← 学习新资料的处理流程
```

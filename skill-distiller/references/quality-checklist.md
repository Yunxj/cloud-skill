# Skill 质量检查清单

生成 Skill 后，逐项检查。有任何 ❌ 的必须修好才能交付。

---

## 一、能用吗（基本可用性）

- [ ] **用户不看 Skill 文件，只说一句指令，AI 就知道该干什么**
  - 如果需要用户先读完一大段说明才能用 → ❌ 太复杂了，简化指令或加示例

- [ ] **输入写清楚了吗**
  - 用户该给什么？给错了怎么办？什么格式？
  - 没写 → ❌ 加上输入说明和格式要求

- [ ] **输出写清楚了吗**
  - AI 该出什么？什么格式？多长/多大？
  - 没写 → ❌ 加上输出格式和样例

- [ ] **触发方式明确**
  - 用户说什么话就触发这个 Skill？
  - 太模糊（比如"帮我处理一下"）→ ⚠️ 改成更具体的触发词

## 二、好用吗（用户体验）

- [ ] **步骤不超过 5 个**
  - 超过 5 步 → ⚠️ 考虑拆成多个子 Skill 或简化流程

- [ ] **有具体例子**
  - 给用户一个"照着说就行"的例子
  - 没有例子 → ⚠️ 至少加一个"示例输入"和"示例输出"

- [ ] **不需要用户懂技术**
  - 出现了 API、JSON、正则、环境变量等术语 → ❌ 要么隐藏细节（让 AI 处理），要么换成大白话解释

- [ ] **用户能自己修改规则**
  - 规则写在独立的文件里（references/），不要全塞在主文件里
  - 塞在一起 → ⚠️ 用户改规则时容易改坏其他部分

## 三、稳吗（稳定性）

- [ ] **边界情况处理了**
  - 输入为空怎么办？
  - 输入太长怎么办？
  - 输入格式不对怎么办？
  - 一个都没处理 → ❌ 至少处理"输入为空"的情况

- [ ] **约束条件写全了**
  - 不能做的事列清楚了吗？
  - 必须做的事列清楚了吗？
  - 漏了 → ⚠️ 补上

- [ ] **不会跟其他 Skill 冲突**
  - 触发词是否太通用？（比如"帮我写"可能跟很多 Skill 撞车）
  - 太通用 → ⚠️ 加上具体场景限定

## 四、能长大吗（可迭代性）

- [ ] **内置了使用记录机制**
  - Skill 文件末尾有"用完记一笔"的指令
  - 没有 → ❌ 加上

- [ ] **版本号有吗**
  - 文件头部或注释里有版本号
  - 没有 → ⚠️ 加上 v1.0.0

- [ ] **规则容易更新**
  - 规则写在独立文件里，改了不用动主文件
  - 全写在一个文件里 → ⚠️ 建议拆分

---

## 检查结果汇报格式

检查完后，这样告诉用户：

```
检查完了，结果如下：

✅ 通过（X项）
- 输入输出明确
- 有具体示例
- ...

⚠️ 建议优化（X项）
- 步骤有点多（7步），建议拆成两个 Skill
- 触发词"帮我分析"太通用，建议改成"帮我做数据分析"
- ...

❌ 必须修改（X项） → 已帮你修好
- 缺少使用记录机制 → 已在末尾添加
- 边界情况未处理 → 已添加"输入为空时的处理"
- ...
```

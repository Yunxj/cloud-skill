# 学习资料处理流程

> 当用户说"把这个资料学一下"、"看看这篇文章有没有用"、"把这个 Skill 的做法学过来"时触发。

---

## 处理步骤

### 1. 接收资料

确认用户给的是什么：

| 类型 | 例子 | 怎么读 |
|---|---|---|
| 文章链接 | 微信公众号、知乎、博客 | `web_ext_scraper` 或 `curl` |
| 本地文件 | .md / .txt / .pdf | `read_file` |
| Skill 文件 | SKILL.md | `read_file` |
| GitHub 仓库 | github.com/xxx/xxx | `gh` 或 `curl` |
| 视频链接 | B站、YouTube | 提取字幕 |

### 2. 阅读和理解

快速通读，提取以下信息：

- **方法论**：有没有新的创建/管理/迭代 Skill 的方法？
- **好模板**：有没有值得学习的 Skill 结构或模板格式？
- **好话术**：有没有跟用户沟通的好表达方式？
- **新路线**：有没有我们 5 条路线之外的创建方式？
- **新案例**：有没有好的 Skill 案例？
- **踩坑经验**：有没有什么"千万别这样做"的教训？

### 3. 判断价值

- **有价值** → 继续第4步
- **跟现有内容重复** → 告诉用户"这个我们已经有类似的了，在 references/xxx.md"
- **价值不大** → 告诉用户"这篇主要讲的是 xxx，跟做 Skill 关系不大。不过我记住里，以后可能有参考价值"

### 4. 更新文件

根据学到的内容，更新对应的文件：

| 学到了什么 | 更新哪个文件 |
|---|---|
| 新方法/新路线 | SKILL.md（路线部分）+ 新建 references/route-xxx.md |
| 好模板结构 | templates/ 对应模板 |
| 好话术 | 对应 references/ 路线文件中的"跟用户怎么说"部分 |
| 踩坑经验 | quality-checklist.md（新增检查项） |
| 好案例 | references/cases.md（如果存在） |
| 跟 Skill 质量相关 | quality-checklist.md |
| 其他 | references/learning-notes.md |

**更新原则**：
- 不覆盖原有内容，只追加或优化
- 保持大白话风格
- 任何更新都要在 learning-notes.md 里记一笔

### 5. 版本更新

如果更新了重要内容（不是仅记录学习笔记），版本号递增：
- 小改动（措辞优化、补了个例子）→ patch：1.0.0 → 1.0.1
- 中改动（加了新检查项、新子步骤）→ minor：1.0.0 → 1.1.0
- 大改动（加了新路线、改了核心流程）→ major：1.0.0 → 2.0.0

更新 SKILL.md 头部的 version 字段和 CHANGELOG.md。

### 6. 汇报

告诉用户学到了什么、改了哪里：

```
学完了！这篇【资料标题】还挺有价值的。

📖 学到了什么：
- 【要点1】
- 【要点2】

🔧 更新了哪里：
- references/xxx.md：新增了...
- quality-checklist.md：加了检查项...
- 版本：v1.0.0 → v1.1.0

你可以继续丢资料给我学，学得越多我越懂行 😎
```

---

## 特别说明

- 学习是增量式的，不会删掉以前学到的内容
- 用户反馈（说"这个 Skill 不好用"/"这里应该这样"）也视为学习输入，用同样的流程处理
- 如果用户说"别学了"/"这篇不用了"，直接停下
